# BuildMap

BuildMap — монорепозиторий системы внутренней навигации по зданиям. Проект состоит из:

- **`api`** — backend на **Spring Boot 3 / Java 21**;
- **`admin_panel`** — административная панель на **React** для управления зонами, этажами, точками маршрута и пользователями;
- **`buildmap_nav`** — пользовательский frontend для построения маршрутов по QR-коду или прямой ссылке;
- **`docker-compose.yml`** — сборка всей системы в три контейнера.

Проект ориентирован на сценарий, в котором администратор или владелец зоны заносит в систему план здания, этажи и точки навигации, а конечный пользователь получает маршрут внутри здания через веб-интерфейс.

## Что умеет система

### Для администратора

- аутентификация через Telegram;
- просмотр подробного дашборда;
- управление пользователями;
- управление всеми зонами (`mapping areas`);
- управление всеми этажами;
- управление всеми навигационными точками (`fulcrums`);
- создание связей между точками любых зон;
- мягкое удаление и force-delete для ряда сущностей;
- генерация PDF с QR-кодами для точек навигации.

### Для пользователя

- аутентификация через Telegram;
- просмотр упрощённого дашборда;
- управление своими зонами (`mapping areas`);
- управление своими этажами;
- управление своими навигационными точками (`fulcrums`);
- создание связей между точками своих зон;
- мягкое удаление для ряда сущностей;
- генерация PDF с QR-кодами для точек навигации.

### Для посетителя

- открытие карты по стартовой точке (`fulcrum`);
- выбор конечной точки назначения;
- построение кратчайшего маршрута;
- переключение между этажами;
- отображение пошаговых инструкций;
- переход по QR-коду формата `.../navigation?fulcrum=<id>`.

## Как работает навигация

Backend формирует маршрут по следующей логике:

1. получает стартовую и конечную точки;
2. проверяет, что обе точки принадлежат одной `MappingArea`;
3. загружает все этажи зоны;
4. собирает все `fulcrums` всех этажей;
5. строит граф;
6. валидирует входной запрос;
7. запускает поиск кратчайшего пути по алгоритму Дейкстры;
8. собирает найденный путь в DTO;
9. генерирует пошаговые инструкции.

Дополнительно система формирует человекочитаемые шаги:

- идти прямо;
- повернуть налево/направо;
- развернуться;
- подняться/спуститься на этаж;
- использует ориентиры (`LANDMARK`) для подсказок.

Проект решает не только задачу вычисления пути, но и задачу инструктажа пользователя внутри помещения.

## Аутентификация и авторизация

### Поддерживаемые сценарии входа

#### 1. Telegram login

Основной пользовательский вход — через endpoint:

```http
POST /api/auth/telegram
```

Backend:
- проверяет подпись данных Telegram;
- ищет пользователя по `telegramId`;
- создает его при первом входе;
- выдает JWT-токен.

#### 2. Dev login

Для разработки:

```http
POST /api/auth/dev-login
POST /api/auth/test-login
```

Они работают только при `AUTH_DEV_ENABLED=true` и при совпадении секрета `AUTH_DEV_SECRET`.

### JWT

После успешной аутентификации backend возвращает JWT. Frontend сохраняет токен в `localStorage` и подставляет его в заголовок:

```http
Authorization: Bearer <token>
```

## REST API: основные группы endpoints

### AuthController

```text
POST /api/auth/telegram
POST /api/auth/dev-login
POST /api/auth/test-login
POST /api/auth/logout
GET  /api/auth/me
GET  /api/auth/test-connection
```

### DashboardController

Судя по frontend API-клиенту, используются endpoints:

```text
GET /api/dashboard/admin
GET /api/dashboard/me
```

### UserController

Используется из админ-панели для:

- получения списка пользователей;
- просмотра пользователя;
- создания;
- обновления;
- soft delete / force delete;
- блокировки и разблокировки.

### MappingAreaController

CRUD по зонам картографирования, в том числе получение зон пользователя.

### FloorController

CRUD по этажам, получение этажей зоны, получение этажа по `fulcrum`.

### FulcrumController

Ключевой контроллер предметной области:

- получение точек по этажу;
- получение точек по зоне;
- создание/обновление/удаление;
- управление связями между точками;
- проверка доступности QR;
- выдача PDF с QR-кодами по зоне.

### NavigationController

```text
POST /api/navigation/path
GET  /api/navigation/debug/graph/{areaId}
```

Главный endpoint пользовательской навигации — `POST /api/navigation/path`.

## Frontend: `admin_panel`

Админка построена вокруг маршрутов:

- `/login`
- `/dashboard`
- `/areas`
- `/floors`
- `/users`

В приложении есть:

- `ProtectedRoute` — защита от неавторизованного доступа;
- `AdminRoute` — защита страниц только для администраторов;
- `PublicRoute` — ограничение доступа к экрану логина для уже вошедших пользователей.

### Основной сценарий

Посетитель открывает маршрут по URL из Qr-кода:

```text
/navigation?fulcrum=<startId>&to=<endId>
```

Где:
- `fulcrum` — стартовая точка;
- `to` — целевая точка.

### Что делает экран Navigation

- читает query-параметры;
- загружает данные по стартовой точке и этажам;
- предлагает поиск конечной точки;
- запрашивает маршрут у backend;
- отображает:
  - активный этаж,
  - точки маршрута,
  - сегменты маршрута,
  - панель шагов,
  - переключение этажей.

### Особенности UX

- поиск исключает стартовую точку и точки типа `CORRIDOR`;
- результаты сортируются по этажу и имени;
- при выборе шага маршрут может фокусироваться на нужном месте карты;
- при межэтажном переходе показываются инструкции «поднимитесь» / «спуститесь».

### Что важно понимать

`buildmap_nav` — это не просто отрисовка SVG, а UI, который работает поверх графовой модели здания. Его задача — синхронизировать:

- текущий этаж;
- выбранную цель;
- список шагов;
- фокус карты;
- найденный путь.

## QR-коды и PDF

Backend-сервис `QrPdfService`:

- генерирует QR с помощью ZXing;
- собирает PDF через PDFBox;
- подгружает шрифты из `resources/fonts`;
- формирует ссылку вида:

```text
<QR_BASE_URL>/navigation?fulcrum=<id>
```

### Endpoint

```http
GET /api/fulcrum/area/{areaId}/qr.pdf?sizeMm=140
```

### Содержимое PDF-страницы

Для каждой точки генерируется отдельная страница с:

- QR-кодом;
- названием точки;
- подписью этажа;
- ID точки.

Это удобный operational workflow: распечатать QR, разместить на объекте, дать пользователю вход в навигацию.

## CI-CD

### workflow:
- Запускается при push в ветку master и состоит из двух джоб: build → deploy.
- Билдит и пушит 3 Docker-образа: buildmap-api, buildmap-admin, buildmap-nav.
- Для фронтендов при сборке прокидывает build args с URL API.
- После сборки по SCP копирует docker-compose.yml на сервер.
- Затем по SSH создаёт .env, подтягивает свежие образы и поднимает проект через docker compose up -d.
- В конце чистит неиспользуемые Docker-образы.

## Запуск через Docker Compose

### 1. Подготовка

Скопируйте пример окружения:

```bash
cp .env.example .env
```

Отредактируйте минимум:

- `DOCKERHUB_USER`
- `TELEGRAM_BOT_TOKEN`
- `QR_BASE_URL`
- при необходимости `AUTH_DEV_ENABLED` / `AUTH_DEV_SECRET`

### 2. Запуск

```bash
docker compose up -d
```

### 3. Доступные сервисы

После запуска по умолчанию:

- API: `http://localhost:8080`
- Admin panel: `http://localhost:3000`
- Navigation UI: `http://localhost:3001`

### 4. Остановка

```bash
docker compose down
```

## Типовой workflow наполнения системы

1. Запустить backend, admin panel и navigation UI.
2. Войти в админ-панель.
3. Создать пользователя или назначить владельца зоны.
4. Создать `MappingArea`.
5. Добавить этажи (`Floor`).
6. Загрузить или вставить SVG-планы этажей.
7. Разместить `Fulcrum`-точки на этажах.
8. Задать типы точек и направления обзора.
9. Создать связи между точками.
10. Проверить маршрут через `buildmap_nav`.
11. Сгенерировать QR-коды и разместить их в реальном пространстве.
